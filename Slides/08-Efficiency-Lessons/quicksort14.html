Quicksort: run time for the best case

The best case for quicksort is when the two lists returned by the call to partition are about the same length, which implies both are about half the length of the given list. If that happens on each recursive call, the lengths of the list for each call are as shown in the following tree, which is an abstraction of the computation graph.

Each node in the tree represents a call to quicksort, passing a list whose length is indicated by the label on the node. Each of those calls takes time roughly proportional to the length of the list, not including the recursive calls. That means the total time is roughly proportional to the sum of all labels in the tree.

The height of the tree is Θ(lg n), and the labels at each level of the tree add up to n, so the best-case running time for quicksort is Θ(n lg n).

That is also the average-case running time, assuming the lists to be sorted are in random order.
